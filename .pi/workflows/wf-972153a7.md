### PVM's core phase primitive (pvm.lua lines 375-850):

```lua
function pvm.phase(name, handlers_or_fn, opts)
    -- Creates a recording triplet boundary.
    -- On hit:  seq_gen over cached array
    -- On miss: recording_gen wrapping handler's output
    -- On repeated miss for same node: share in-flight recording
    
    -- Cache modes: args_cache = "full"|"last"|"none", node_cache = "full"|"none"
    -- Two cache trees per phase: keyed_cache (node → array), keyed_cache_args (node→args→array)
    -- Two pending trees (for in-flight miss deduplication)
    
    -- Returns a callable boundary object. When invoked as phase(node, ...):
    -- 1. Resolve ASDL class and cache key
    -- 2. Check node cache → hit returns seq_n_gen/seq_gen
    -- 3. Check pending (in-flight) → share recording_gen
    -- 4. Miss: call handler → wrap output in recording_gen
    
    -- Recording entries share state across concurrent consumers:
    --   { cache_parent, pending_parent, slot, buf, n, done, g, p, c, packed, ... }
end
```

### Three distinct "phase" concepts in the codebase:

| Concept | File | Nature |
|---------|------|--------|
| **PVM recording phase** | `pvm.lua` | Lazy cache boundary with recording/fusion. Lua implementation. |
| **Compiler package phase** | `schema/phase.lua`, `compiler_package.lua` | Data declaration of compiler world transitions. Executed by LLBL processes, not PVM. |
| **LLBL process event** | `frontend_pipeline.lua` | `ctx:phase(payload)` — progress events in an LLBL process handle. Not PVM-related. |

### PVM vs LLPVM shared architecture:

| Concept | PVM (Lua) | LLPVM (native) |
|---------|-----------|----------------|
| Phase key | `{cls, key, args}` | `LlPhaseKey {phase, input_identity, args, ...}` |
| Recording | Lua table with REC_* indices | `LlRecording {key, source, buffer, state}` |
| Cache entry | Lua table with hit array | `LlCacheEntry {key, buffer, complete}` |
| Cache tree | Weak-keyed nested Lua tables | Handle-indexed slot arrays |
| Fusion | LuaJIT trace through nested recording gen | Native region graph |
| GC | Lua GC (weak tables) | Explicit allocator + OOM protocol |

## Observations

1. **PVM is an implementation substrate, not the compiler architecture.** The user's initial understanding was correct — PVM provides schema values, interning, phase triplets, and cache boundaries. But the *compiler phase pipeline* (`frontend_pipeline`) is pure procedural composition; it doesn't use PVM recording phases for its own passes.

2. **The compiler_package's "phase" is not a PVM phase.** Despite the name overlap, `compiler_package.lua` declares `phase. typecheck { ... }` using `PhaseDsl`, which produces `LalinPhase.Phase` ASDL records. These are *executed* by `phase_execute.lua`, which delegates to LLBL `process` objects — not PVM recording boundaries. The word "phase" means "a declarative step in the compiler graph", not "a PVM recording boundary."

3. **PVM recording is used by the UI framework, not the compiler.** All `pvm.phase()` calls are in `lua/ui/`. The UI framework uses PVM for layout/rendering phases (compose, measure, resolve, lower, render, interact). The compiler uses PVM only as a type system (ASDL class dispatch, structural update).

4. **`pvm_ll/` is empty scaffolding.** This was probably intended to house concrete LLPVM-backed phase definitions that bridge the PVM Lua implementation and the LLPVM native store. It's currently unused.

5. **The schema layer defines no PVM concepts.** `schema/phase.lua` defines the *phase graph metadata* (worlds, machines, phases, roots). `pvm.phase()` is not defined or referenced anywhere in the schema layer. The ASDL wrapper files the user asked about (`pvm_surface_asdl.asdl_wrapper.lua`) don't exist.

6. **`pvm_erase.lua` confirms the architecture.** It rewrites simple uncached PVM phases into direct function calls, showing that PVM overhead is meaningful enough to require compile-time elimination for hot paths. The erased output depends on `schema_runtime.lua`, a minimal PVM-free alternative for `context()` and `classof()`.

7. **LLPVM mirrors PVM's design at the native level.** Both have the same core concepts (phase, recording, cache entry, phase key, buffer), but LLPVM uses handles and packed C structs while PVM uses Lua weak-keyed tables. The `llpvm.native.*.mlua` files are executable region specifications that compile to C.

8. **PVM is exported as a public module.** `lalin.init` exports `M.pvm = require("lalin.pvm")`, making PVM a first-class part of the public surface. But the `init.lua` also exports `M.schema`, `M.phase_plan`, `M.phase_execute` etc., so both models are available.

9. **79+ files require `lalin.pvm`** — but almost exclusively for `pvm.classof()` and `pvm.context()`, not for phase/cache boundaries. PVM's role as "ASDL infrastructure" dominates its actual usage.

10. **The DESIGN_BIBLE clarifies the intellectual lineage.** PVM is the *generalized insight*: compiler phases as lazy machines. LLPVM is the *concrete canonical form* of that insight at the native boundary. The design evolution is: PVM (Lua, schema values, fusion via LuaJIT traces) → LLPVM (native handles, packed stores, explicit region-based fusion).
