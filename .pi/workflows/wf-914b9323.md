# Residual-Native Hard-Yank Design Artifact

## Goal

Enable Lalin’s native backend to become a residualless copy-patch compiler whose semantics are owned by existing `LalinCode`, `LalinKernel`, and `LalinStencil` ASDL leaves, while machine artifact shape is owned by a new `LalinNative` ASDL schema. The migration is a **hard yank**: delete residual-native concepts, runtime residual C compilation, exact-cell bank semantics, fallback paths, and compatibility wrappers rather than preserving them under renamed APIs.

## Incentives

The current implementation still treats native execution as “residual” work: a bag of deferred compiler activity selected by string modes, represented by `LalinResidual`, and materialized through runtime TCC/C glue or MC/BC paths. This conflicts with `docs/RESIDUAL_NATIVE_ARCHITECTURE.md`, which is binding and explicitly declares the architecture residualless.

The concrete pain points are:

- `lua/lalin/schema/residual.lua` is not only misnamed; it bundles unrelated phase concepts: rejection reasons, lowering targets, C residual units, stencil storage, compile requests, exact MC storage, patch templates, and bank-generation controls.
- `ResidualFunctionPlan`, `CResidual*`, and `StencilRequiresCompile` are active schema concepts even though the target architecture forbids them.
- Runtime native compilation currently includes a residual C/TCC path through `lua/lalin/residual_native.lua`, `lua/lalin/luajit_emit.lua`, and `lua/lalin/c_tcc.lua`; the target architecture says no runtime residual C path exists.
- The current backend has stringly materializer selection via `"mc"`, `"bc"`, and `"tcc"` options, mixing native copy-patch, LuaJIT bytecode, and residual C behavior.
- The MC bank has partially moved toward template manifests and away from embedded exact bytes, but the schema still preserves exact-cell vocabulary and compile-later alternatives.
- Runtime install currently returns ad hoc Lua records such as `{ kind = "InstalledMCStencil", ... }`, violating the ASDL doctrine that compiler/backend facts must be explicit typed ASDL values.
- Tests already encode some target-adjacent invariants, such as no implicit BC fallback and no exact MC byte-array generator output, but they still do so through residual names and APIs.

The hard-yank decision exists to prevent residual vocabulary from surviving as a compatibility layer. The migration should make the native compiler’s architecture explicit, typed, and owned by the correct ASDL schemas.

## Chosen Target

### Approach

The chosen approach is **hard-yank migration only**.

This means:

- Delete residual-native concepts instead of renaming them compatibly.
- Introduce `lua/lalin/schema/native.lua` as the ASDL owner for native artifact shape.
- Remove `LalinResidual` from native compiler paths.
- Remove runtime residual C/TCC compilation from native execution.
- Remove `ResidualFunctionPlan`, `CResidual*`, `StencilRequiresCompile`, exact-cell bank enumeration, fallback native paths, coverage accounting, cap semantics, and budget semantics.
- Replace ad hoc Lua result records with typed `LalinNative` ASDL products and sums.
- Replace internal string dispatch and materializer selection with ASDL leaf methods.
- Keep semantic lowering on existing semantic ASDL leaves, not on native mirror types.
- Preserve only target-valid invariants from current code and tests.

There are no compatibility wrappers. There is no transitional residual facade. Public or internal code that depends on residual-native architecture must be updated or removed.

### Ownership boundaries

#### Semantic owners

Semantic compiler shape remains in existing ASDL schemas:

- `LalinCode`
- `LalinKernel`
- `LalinStencil`

These own all semantic facts about functions, blocks, instructions, terms, places, constants, types, kernels, effects, results, stencil producers, accesses, point expressions, bodies, sinks, schedules, ABIs, and proofs.

NativeCopyPlan:install_native(input)
  -> allocate executable memory
  -> copy selected NativeCompiledTemplate text sections
  -> validate patch bindings

  -> produce NativeExecutable
  -> return NativeInstallSucceeded or NativeInstallRejected

### Error model

The native compiler has exactly four failure classes:

1. **User semantic diagnostic before native compilation**  
   These belong to earlier semantic phases, not `LalinNative`.

2. **Bank-build reject while producing template artifacts**  
   Represented by `NativeTemplateBankBuildRejected` and `NativeTemplateBuildReject` leaves.

3. **Install reject while patching executable code**  
   Represented by `NativeInstallRejected` and `NativeInstallReject` leaves.

4. **Missing ASDL method as hard internal error**  
   This is not a normal result value. The compiler does not create a “not implemented” or “requires compile” artifact.

No schema value may represent missing compiler architecture as normal control flow.

### Tradeoffs acknowledged

The hard yank sacrifices incremental compatibility. Existing residual APIs, option names, test names, generated comments, and helper modules will break rather than being shimmed.

This is acceptable because compatibility wrappers would preserve the core architecture bug: residual as an undefined compiler-work bag. The target requires ASDL-first native facts, concrete leaf methods, typed build/install results, and no runtime residual C path. A compatibility layer would make future agents preserve forbidden concepts and would keep semantic/backend ownership unclear.

The hard yank also increases immediate migration cost. `lua/lalin/schema/residual.lua`, `lua/lalin/residual_native.lua`, `lua/lalin/residual_mc.lua`, `lua/lalin/luajit_backend.lua`, `lua/lalin/init.lua`, generated bank tooling, emitted artifacts, and tests will all be affected. This cost is intentional: the migration is architectural, not cosmetic.

### Risks acknowledged

Known risks from the prior analysis:

- Valuable copy-patch mechanics are mixed into `residual_native.lua`; deleting residual concepts must not accidentally discard target-valid patch projection/application behavior.
- Current MC bank tooling relies on GCC and ELF/readelf details. Those assumptions must be represented cleanly in native template build products or kept as implementation detail behind `NativeTemplateSource:compile_native_template`.
- Runtime install currently uses informal Lua records for memory, base pointers, size, RWX policy, and endian checks. These real invariants must become typed install/runtime facts where they are compiler-visible.
- The public facade currently exposes residual options. Removing them will require coordinated updates to tests and callers.
- The BC path is currently framed as a residual materializer. It must not remain as a residual umbrella over native and non-native artifacts.
- Error handling currently mixes `nil, err` Lua boundary returns with residual rejection ASDL values. The migration must keep boundary errors deliberate while internal native failures use typed native result sums.
- Existing documentation outside `docs/RESIDUAL_NATIVE_ARCHITECTURE.md` may still say MC falls back to BC or describe residual MC/BC as active architecture. The binding decision overrides those stale references.
- Tests with residual names may currently protect useful invariants. They must be rewritten, not blindly deleted, where the invariant remains target-valid.

## Summary Rule for Future Agents

When native implementation pressure appears, do not add residual plumbing, string modes, side tables, fallback paths, or Lua result records. Add the missing `LalinNative` ASDL product, sum, field, template family, patch coordinate, patch hole, graph edge, protocol, build reject, install reject, or executable result, then install the method on the concrete ASDL leaf that owns the behavior.

- LuaJIT-specific MC bank shape as the native bank model;
- residual MC naming as public/internal architecture;
- installed native code represented by ad hoc Lua records;
- exact-cell bank enumeration;
- runtime object extraction as native materialization;
- residual materializer dispatch;
- C/TCC/runtime compilation fallback;
- treating unsupported native coverage as a residual compile-later path.

Any useful implementation detail from `residual_mc.lua` may survive only inside offline generation/build tooling and only if its output is typed `LalinNative` embedded-bank data.

### Tradeoffs acknowledged

This decision sacrifices developer convenience: native bank generation becomes an explicit offline/build step, and runtime can no longer compile missing templates on demand.

That sacrifice is acceptable because it enforces the hard runtime boundary required by the residualless architecture. Runtime native execution becomes deterministic import/copy/patch/install over typed ASDL artifacts instead of a mixed compiler-toolchain path.

It also increases pressure on the generator: all symbol, relocation, section, target, and hole-layout correctness must be validated before or during embedded-bank import. Runtime cannot paper over incomplete embedded data by recompiling.

### Risks acknowledged

Known risks carried forward:

- The embedded representation may initially miss facts that old `residual_mc.lua` inferred implicitly from object files or LuaJIT artifact shapes.
- ELF/GCC/readelf assumptions still exist, but they are now confined to offline tooling and must not leak into runtime semantics.
- Import validation must be strong enough to catch malformed generated banks before install.
- Tests currently named around residual MC may protect useful invariants and must be rewritten carefully rather than blindly deleted.
- `luajit_backend.lua` and `init.lua` still contain residual public/API assumptions and must be updated consistently with the offline-bank model.
- Runtime install invariants such as memory permissions, endianness, patch ranges, relocation compatibility, and executable layout must be represented as typed native facts/results where compiler-visible.

## Summary Rule

For T007 and all follow-on native work: **offline tools may build embedded native template banks; runtime may only import, select, copy, patch, install, and call typed native ASDL artifacts.** No residual MC layer, runtime GCC/readelf/TCC, compatibility shim, exact-cell bank, or ad hoc Lua install record is part of the target architecture.

## Pattern Memo

- ASDL first; Lua methods explain typed ASDL values.
- Concrete ASDL leaves own behavior.
- C stencils only; no assembly template source.
- Runtime only imports/copies/patches/installs; no runtime compiler/tools.
- No residual, fallback, exact-cell, coverage, cap, or compatibility shim.
- No side tables, handler maps, kind/class dispatch, or Lua `{ kind = ... }` records.
- Missing architecture is hard internal error; build/install rejects are only real validation failures.
