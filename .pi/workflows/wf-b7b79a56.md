## Key Code

### PUC opcode contracts

`lopcodes.h` defines the relevant opcodes and notes:

```c
OP_GETTABLE,/* A B C R[A] := R[B][R[C]] */
OP_SETTABLE,/* A B C R[A][R[B]] := RK(C) */
OP_MMBIN,/* A B C call C metamethod over R[A] and R[B] */
OP_LEN,/* A B R[A] := #R[B] */
OP_CONCAT,/* A B R[A] := R[A].. ... ..R[A + B - 1] */
OP_CALL,/* A B C R[A], ... ,R[A+C-2] := R[A](...) */
OP_TAILCALL,/* A B C k return R[A](...) */
OP_RETURN,/* A B C k return R[A], ... ,R[A+B-2] */
OP_TFORPREP, OP_TFORCALL, OP_TFORLOOP
OP_SETLIST,/* A vB vC k R[A][vC+i] := R[A+i], 1 <= i <= vB */
OP_VARARG,/* A B C k R[A], ..., R[A+C-2] = varargs */
OP_GETVARG, /* A B C R[A] := R[B][R[C]], R[B] is vararg parameter */
OP_ERRNNIL,/* A Bx raise error if R[A] ~= nil ... */
```

Important notes in same file:

- CALL: `B == 0` uses previous `top`; `C == 0` sets `top` for next open instruction.
- VARARG: `C == 0` uses actual vararg count and sets top; `k` means vararg table in `R[B]`.
- RETURN: `B == 0` returns up to `top`.
- SETLIST: `B == 0` uses `top`; `k` extends `C` via following EXTRAARG.
- RETURN/TAILCALL `k` signals upvalue closing; `C > 0` signals hidden varargs.

### PUC GETTABLE/SETTABLE

`lvm.c` fast path plus metamethod chain:

```c
vmcase(OP_GETTABLE) {
  TValue *rb = vRB(i);
  TValue *rc = vRC(i);

    luaV_fastgeti(rb, ivalue(rc), s2v(ra), tag);

`luaV_finishget`:
- non-table → `__index` or type error
- table miss without `__index` → nil

## Rejected Old Assumptions

The reset explicitly rejects these assumptions:

1. **Protocol out-tags are not success.**

2. **Host/interpreter handoff is not MoonOut compilation.**  
   A compiled kernel may not rely on a Lua interpreter, opcode helper, resume protocol, or external dispatcher to finish Lua semantics.

3. **Rejection is not success for valid Lua behavior.**  
   Unsupported valid behavior may remain unimplemented, but tests must not count rejection as a positive compiled result.

4. **Primitive opcode patches are frozen until design is settled.**  
   Individual fixes to `ADD`, `GETTABLE`, `SETLIST`, `VARARG`, etc. must not proceed without the flat region-command target and continuation policy.

5. **MoonOut may not remain a wrapper around `LuaNF.Program`.**  
   The output layer must become a terminal command layer.

## Chosen Target

### Approach

The selected design is the **Typed Moonlift Region Target**.

MoonOut becomes a flat list of commands. Each command emits or references a typed Moonlift semantic region. Generic Lua behavior is implemented inside compiled Moonlift regions, not by returning protocol tags to the host.

This choice preserves the Moonlift philosophy: typed control edges, explicit continuations, zero-cost region splicing via `emit`, and no integer continuation protocol.

### Architecture

#### Layer shape

MoonOut must be reshaped around a flat command product.

The exact ASDL syntax remains to be finalized before coding, but the required shape is:

```asdl
module MoonOut {
    Kernel = (
        MoonOut.KernelKind kind,
        MoonOut.Param* params,
        MoonOut.Cmd* cmds,
        ...
    ) unique

    Cmd = (
        MoonOut.CmdKind kind,
        MoonOut.RegionRef region,
        MoonOut.Operand* args,
        MoonOut.ContBinding* continuations,
        MoonOut.Obligation* obligations
    ) unique
}
```

The important invariant is not the exact field names; it is that `Kernel` carries flat `Cmd*`, not `LuaNF.Program`.

`Cmd` must be one uniform product shape. Semantic variation belongs in ASDL-visible fields such as `kind`, `region`, operands, continuation bindings, and obligations, not in hidden Lua closures or opaque string protocols.

#### Meaning of a MoonOut flat command

A `MoonOut.Cmd` means:

> Emit this typed Moonlift semantic region with these operands and these typed continuations.

It is not:

- a PUC opcode,
- a Lua interpreter instruction,
- a host callback request,
- a protocol exit record,
- a string of Moonlift source carrying hidden semantics.

A command is valid only if the referenced region fully implements the semantic operation it claims to represent, or if it represents a legitimate typed kernel boundary such as a return/error/guard boundary according to the continuation rules.

#### Typed region requirements

A Moonlift region used by MoonOut must satisfy:

- It is a compiled Moonlift `region` or equivalent typed fragment.
- Its parameters are typed data parameters.
- Its exits are typed continuations.
- All continuation edges are explicit at the `emit` site.
- Its contract is represented in ASDL-visible data.
- It does not call back into the Lua interpreter to complete semantics.
- It does not encode control as integer tags plus mutable out-params.
- It may be reusable, but reuse must be through typed region composition, not opaque helper dispatch.

Examples of region families that may eventually exist under this model include table get/set, arithmetic metamethod handling, concat, len, call preparation, vararg adjustment, close chains, and generic-for behavior. They are legal only when they fully implement the relevant Lua semantics for their claimed domain.

#### Continuation rules

Continuations such as `ok`, `error`, `yield`, `guard`, branch, return, or other semantic exits must be typed Moonlift continuations and resolved inside the compiled kernel boundary.

They must not become:

```text
out_tag = call
out_tag = setlist
out_tag = getvarg
out_tag = generic_for
```

or any equivalent host protocol under another name.

If a kernel exposes a boundary result, that boundary must be typed and explicit in Moonlift control/data terms. It must not be an untyped integer tag protocol requiring an external interpreter dispatcher.

#### Legal helpers

Legal helper-like reuse is limited to compiled, typed Moonlift composition:

- `region` definitions emitted or referenced by MoonOut;
- `emit` sites with explicit continuation bindings;
- typed fragments whose semantics and obligations are ASDL-visible;
- primitive operations that are truly primitive for the backend and not hidden Lua semantic interpreters.

#### Illegal helpers

The following are illegal in accepted MoonOut kernels:

- Lua interpreter handoff;
- opcode helper calls such as old `OpcodeHelper`/`HelperCall` patterns;
- host callbacks that implement Lua semantics outside the compiled kernel;
- integer `out_tag` protocol exits;
- mutable `out_*` pointer protocols as semantic continuation ABI;
- side tables that carry semantic meaning not represented in ASDL;
- tests that assert rejection or protocol handoff as success.

### Data flow after reset

The intended data flow becomes:

```text
LuaSrc.Window + LuaEvidence
  -> LuaSem / semantic lowering
  -> LuaNF / normalized semantics where still useful
  -> LuaContract
  -> MoonOut lowering
       produces MoonOut.Kernel(cmds = flat Cmd*)
  -> MoonOut validation
       proves command/region/continuation closure
  -> MoonOut emission
       for _, cmd in ipairs(kernel.cmds) do
           emit typed Moonlift region usage
       end
```

The final boundary is `MoonOut.Cmd*`, not `LuaNF.Program`.

### Tradeoffs acknowledged

This design sacrifices some transparency compared with a primitive microcode model: generic Lua semantics may live inside reusable typed regions rather than being expanded into tiny primitive commands.

That is acceptable because Moonlift’s native abstraction is typed region composition. The critical requirement is that regions are compiled semantic splices with typed continuations, not opaque helpers or interpreter protocols.

This design is more invasive than merely banning protocol exits because it requires replacing the pointer/tag ABI and defining region contracts. It is still preferred because it supports fuller Lua semantics without building a second hand-written Lua VM microcode language.

### Risks acknowledged

Known risks from critique:

- Typed regions can become disguised helpers if their contracts and continuation edges are not explicit and validated.
- MoonOut may become coupled to a semantic region library.
- Proving that a region fully implements Lua semantics is harder than structurally banning protocol tags.
- Migration cost is higher than the minimal honest-partial reset.
- Existing runtime modules contain useful semantic fragments but also known discrepancies; importing them blindly could reintroduce interpreter machinery or wrong semantics.

The design therefore requires strict validation and design-first implementation discipline.

   Rejecting valid Lua behavior may be the current implementation limitation, but tests must not count it as successful compilation.

7. **MoonOut wrapping LuaNF**  
   `MoonOut.Kernel` / `MoonCFG.Kernel` must not carry `LuaNF.Program` as the executable payload.

## How Generic Lua Operations Compile

Generic Lua operations must become CFG regions/fragments, not protocol exits.

### Arithmetic / MMBIN

Generic arithmetic must model the PUC arithmetic + `MMBIN*` pair as one semantic unit when necessary.

Compiled CFG may include:

- primitive numeric fast path blocks,
- type-choice branches,
- metamethod lookup blocks,
- metamethod-call region,
- result projection blocks,
- error exits.

The `MMBIN*` marker must not become a standalone no-op unless the primitive-specialized arithmetic window proves that the marker is unreachable semantically.

### GETTABLE / SETTABLE

Generic table access must become CFG regions for:

- table raw get/set,
- array/index fast paths,
- field/const-key paths,
- `__index` / `__newindex` lookup,
- table-chain continuation,
- function metamethod call,
- loop-limit/type-error exits.

It must not become `out_tag = gettable` or any equivalent host request.

### LEN

Generic `LEN` must become CFG covering:

- string byte length,
- table raw length when no `__len`,
- `__len` metamethod lookup/call,
- type-error exit.

Current table-shape-only lowering remains a specialization, not full generic coverage.

### CONCAT

Generic `CONCAT` must become CFG covering:

- string operands,
- numeric-to-string conversion where valid,
- `__concat` lookup/call,
- continuation after metamethod result,
- error paths.

Current all-operands-string lowering remains a specialization.

### CALL / TAILCALL

Generic call behavior must become CFG covering:

- function classification,
- `__call` metamethod retry,
- argument/result count handling,
- tailcall behavior,
- upvalue close behavior when required,
- typed return/error/yield continuations.

It must not become `CallProtocolExit`.

### RETURN

Return semantics must preserve and compile the relevant PUC fields:

- fixed count,
- open count via `top`,
- `C`,
- `k` close behavior,
- hidden vararg behavior where applicable.

Multivalue/open returns are valid Lua behavior and must not be counted green by rejection.

### VARARG / GETVARG

Vararg behavior must become CFG over explicit frame/vararg places:

- fixed-count copy,
- open-count copy and `top` update,
- vararg table behavior,
- `GETVARG` lookup semantics.

It must not become `GetVargProtocolExit`.

### SETLIST

`SETLIST` must become CFG covering:

- fixed value count,
- open count from `top`,
- `k` / `EXTRAARG` extension,
- array write loop,
- table/barrier semantics as required by the chosen runtime model.

It must not become `SetListProtocolExit`.

### CLOSE / TBC

Close behavior must become CFG covering:

- close-chain traversal,
- `__close` lookup,
- close-call sequencing,
- error/unwind behavior.

It must not become `CloseProtocolExit`.

### Generic For

`TFORPREP`, `TFORCALL`, and `TFORLOOP` must become CFG covering:

- iterator/state/control layout,
- iterator call,
- result placement,
- nil test,
- loop continuation,
- close behavior where applicable.

It must not become `GenericForProtocolExit`.

## Emission Rules

`moon_out_emit.lua` must become a renderer from `MoonCFG` to Moonlift source.

Allowed emitter responsibilities:

- assign stable source names for ASDL identifiers;
- render types, params, blocks, regions, expressions, operations, and terminators;
- render `emit` sites for ASDL-described regions;
- render typed continuations exactly as represented in ASDL;
- preserve deterministic output.

Forbidden emitter responsibilities:

- inspect `LuaSrc.Op` to choose semantics;
- interpret `LuaNF.Exit` as executable behavior;
- lower protocol exits into tags;
- synthesize hidden helper calls;
- encode semantic choices as strings or integer tags;
- decide metamethod/table/call semantics during printing.

The semantic lowering decision must already be present in `MoonCFG`.

- Accepted compile product is `MoonCFG.Kernel`, not `MoonOut.Kernel`.
