# VALUE_LAYOUT.md — Moonlift VM Value and Object Layouts

Status: canonical data layout reference. Based on deep-dive of lj_obj.h, lj_gc.c, lj_dispatch.h.
Audience: VM implementors.

---

## 1. TValue — NaN-Boxed Tagged Value

### 1.1 The fundamental insight

x64 processors use only 48 bits for virtual addresses (bits 0-47). Bits 48-63
must be copies of bit 47 (sign-extended canonical form). This leaves the top
17 bits free for type tags when storing pointers.

The canonical x64 NaN (generated by FPU) is `0xfff8_0000_0000_0000`. ANY
64-bit value with the MSW > `0xfff80000` is a "special NaN" — the FPU will
never produce it, and it can be used to carry type information.

### 1.2 Layout

```
TValue: u64 (8 bytes)

Non-GC64 mode (32-bit GC refs):
  [63:32] it: u32    — type tag (overlaps MSW of double)
  [31:0]  value      — GCRef, int32, or low bits of double

GC64 mode (64-bit GC refs):
  [63:47] tag     — 17 bits of NaN payload (all 1s + 4-bit type)
  [46:0]  pointer — 47-bit GC pointer / lightuserdata
  ... or [31:0] int32 — zero-extended integer

LJ_FR2 mode (frame type/size in TValue):
  [63:0] ftsz     — frame type and size as i64
```

### 1.3 Type tags

Tags are negated small integers in the MSW — this makes type checks a single
signed comparison against an immediate:

```
LJ_TNIL     = ~0u  = 0xFFFFFFFF
LJ_TFALSE   = ~1u  = 0xFFFFFFFE
LJ_TTRUE    = ~2u  = 0xFFFFFFFD
LJ_TLIGHTUD = ~3u  = 0xFFFFFFFC
LJ_TSTR     = ~4u  = 0xFFFFFFFB
LJ_TTAB     = ~11u = 0xFFFFFFF4
LJ_TFUNC    = ~8u  = 0xFFFFFFF7
LJ_TNUMX    = ~13u = 0xFFFFFFF2  — canonical number marker
```

**Why negated?** On x64, `cmp [reg], imm8` with sign-extended 8-bit immediate
is a compact 4-byte instruction. `~0u` through `~13u` all fit in 8 bits with
sign extension. When the MSW is `0xFFFFFFF-` the comparison `it >= LJ_TISNUM`
(where LJ_TISNUM = ~13u = 0xFFFFFFF2) becomes a single unsigned comparison.

**Type ranges (ORDER LJ_T):**
- `LJ_TISPRI = LJ_TTRUE` — nil/false/true are first three
- `LJ_TISNUM = LJ_TNUMX` — number (integer or double) threshold
- `LJ_TISGCV = LJ_TSTR+1` — all GC objects are after strings

The ordering is INTENTIONAL. `tref_typerange` checks like `tref_ispri(tr)`
compile to a single unsigned range comparison `ref_type - first <= last-first`.

### 1.4 Explicit-tag representation (Moonlift bring-up)

For the first bring-up, we use an explicit tag/payload representation
to avoid the complexity of NaN-boxing arithmetic:

```moonlift
struct TValue
    tag: i32
    _pad: i32
    payload: i64
end
```

After the VM is working, NaN-boxing can replace this behind a helper region
API. The explicit-tag fields are designed for the eventual NaN-boxing layout:
`tag` at offset 0 maps to the MSW of the 64-bit TValue.

## 2. GCHeader — Common Object Prefix

```
GCHeader: 8 bytes
  [0:4] or [0:8] nextgc: GCRef  — intrusive GC list (4 or 8 bytes depending on GC64)
  [4:5] marked: u8               — GC color + finalization flag
  [5:6] gct: u8                  — object type tag (see §2.2)

Remark: "This occupies 6 bytes, so use the next 2 bytes for non-32 bit fields."
         — Mike Pall, lj_obj.h
```

### 2.1 GCRef — the pointer tag trick (GC64 mode)

In GC64 mode, `setgcreft(r, gc, it)` stores:
```c
r.gcptr64 = (uint64_t)&(gc)->gch | (((uint64_t)(it)) << 47)
```

The object pointer uses bits 0-46. Bits 47-63 carry the internal tag `it`.
This is the same NaN-boxing trick used for TValues.

### 2.2 GC type tags (gct field)

```
GCT_STR    = 0   — interned string
GCT_UPVAL  = 1   — upvalue
GCT_THREAD = 2   — coroutine thread
GCT_PROTO  = 3   — function prototype
GCT_FUNC   = 4   — Lua or C function (disambiguated by flags)
GCT_TRACE  = 5   — compiled trace
GCT_CDATA  = 6   — FFI cdata
GCT_TAB    = 7   — hash table
GCT_UDATA  = 8   — userdata
```

### 2.3 Mark color bits (white0/white1 alternation)

```
white0 = 0  — current white (bit 0)
white1 = 1  — other white (bit 0 + mix)
gray   = 2  — being scanned (bit 1 + mix)
black  = 3  — scanned, no white refs (bits 1+0 set)
```

White colors ALTERNATE (white0/white1) to avoid a full sweep of all objects.
`currentwhite` in GlobalState tracks which white is "current". Objects from
the previous GC cycle have the other white value and are treated as garbage.
On each full GC cycle, `currentwhite` flips.

## 3. GC Dead Key Design — The Critical Non-Obvious Decision

**LuaJIT intentionally does NOT remove dead keys from tables during GC.**
This is a deliberate departure from standard Lua.

Rationale: Removing dead keys would change hash slot assignments when a value
flips between nil and non-nil between GC cycles. This would cause HREFK
(hash-reference-by-constant-key) to back off, invalidating compiled traces.
Stable hash slots mean table access traces stay valid across GC cycles.

Trade-off: Write barriers for table stores must check BOTH key and value for
GC colors. Dead keys accumulate until a table resize (NEWREF). The memory
cost is bounded by table utilization.

From lj_obj.h: "ALL uses of lj_tab_set* require a barrier for the stored value
*and* the stored key, based on the above rules."

## 4. GC Object Layouts

### 4.1 GCstr — Interned String

```
GCstr:
  [0:8]   GCHeader
  [8:9]   reserved: u8    — lexer reserved-word flag
  [9:10]  hashalg: u8     — hash algorithm
  [10:12] sid: u32        — interned string ID (enables O(1) equality)
  [12:16] hash: u32       — precomputed hash value
  [16:20] len: u32        — byte length (excludes NUL)
  [20:..] data: [len+1]u8 — inline bytes + NUL terminator
```

### 4.2 GCtab — Hash Table

```
GCtab:
  [0:8]   GCHeader
  [8:16]  array: ptr(TValue)    — array part (size = asize)
  [16:20] asize: u32            — array size (power of 2)
  [20:24] hmask: u32            — hash mask (numnodes - 1)
  [24:32] node: ptr(Node)       — hash nodes
  [32:36] nomm: u32             — negative metamethod cache (bitmask)
  [36:40] fastkey: u32          — fast lookup hint
  [40:48] metatable: ptr(GCtab) — metatable
```

**Node layout (24 bytes):**
```
  [0:8]   val: TValue      — stored value
  [8:16]  key: TValue      — key
  [16:24] next: ptr(Node)  — hash chain link
```

### 4.3 GCproto — Function Prototype

```
GCproto:
  [0:8]   GCHeader
  [8:16]  bc: ptr(BCIns)    — bytecode array
  [16:20] sizebc: u32       — bytecode instruction count
  [20:28] k: ptr(TValue)    — constant table
  [28:29] numparams: u8     — fixed parameter count
  [29:30] framesize: u8     — minimum frame size in slots
  [30:31] flags: u8         — vararg + ILoop flags
  [31:32] sizekgc: u8       — number of GC constants
  [32:36] trace: i32        — trace number (0 if none)
```

### 4.4 GCfuncL — Lua Function

```
GCfuncL:
  [0:8]   GCHeader
  [8:12]  nupvalues: u8       — number of upvalues
  [12:16] ffid: u8            — 0 for Lua functions
  [16:24] proto: ptr(GCproto) — function prototype
  [24:..] upvals: [nupvalues]ptr(GCupval)
```

### 4.5 GCfuncC — C Function

```
GCfuncC:
  [0:8]   GCHeader
  [8:12]  nupvalues: u8       — number of upvalues (max 2)
  [12:16] ffid: u8            — fast-function ID (non-zero)
  [16:24] fn: ptr(u8)         — C function pointer
  [24:..] upvals: [nupvalues]TValue  — embedded upvalue storage
```

### 4.6 GCupval — Upvalue

```
GCupval:
  [0:8]   GCHeader
  [8:16]  v: ptr(TValue)        — pointer to real value (in stack, or &closed)
  [16:24] closed: TValue        — closed copy
  [24:32] open_next: ptr(GCupval)
  [32:40] open_prev: ptr(GCupval)
```

Open upvalues form a doubly-linked list rooted in L->openupval. When a
stack slot goes out of scope, the upvalue is "closed": the value is copied
into `closed`, and `v` is redirected to `&closed`.

### 4.7 GCthread — Lua Thread/Coroutine

```
GCthread:
  [0:8]   GCHeader
  [8:16]  stack: ptr(TValue)       — stack memory
  [16:20] stacksize: u32           — allocated slots
  [20:24] top: ptr(TValue)         — current top-of-stack pointer
  [24:28] base: ptr(TValue)        — current frame base pointer
  [28:32] maxstack: u32            — max stack depth reached
  [32:40] pc: ptr(BCIns)           — instruction pointer
  [40:44] status: i32              — LUA_OK, LUA_YIELD, LUA_ERRRUN, etc.
  [44:48] glref: GCRef(GlobalState) — backward pointer to global state
  [48:..] openupval: ptr(GCupval)  — open upvalue list
```

## 5. GlobalState

```
GlobalState:
  allgc: ptr(GCobj)        — head of all-GC-objects list
  gray: ptr(GCobj)         — gray object list
  grayagain: ptr(GCobj)    — gray-again list (atomic barrier)
  weak: ptr(GCobj)         — weak table list
  gcstate: u32             — current GC phase
  currentwhite: u32        — active white color (0 or 1, alternates)
  totalbytes: u32          — total allocated bytes
  gcdebt: u32              — GC "debt" (trigger threshold)
  gcstepmul: u32           — step multiplier
  mainthread: ptr(GCthread) — main Lua thread
```

## 6. BCIns — Bytecode Instruction

```
BCIns: u32

Layout A (8+8+8+8):
  [7:0]   op: u8   — opcode
  [15:8]  A:  u8   — result slot
  [23:16] C:  u8   — operand slot / literal
  [31:24] B:  u8   — operand slot / literal

Layout D (8+8+16):
  [7:0]   op: u8   — opcode
  [15:8]  A:  u8   — result slot
  [31:16] D: u16   — literal operand

Accessors:
  bc_op(ins)  =  ins & 0xFF
  bc_a(ins)   = (ins >>  8) & 0xFF
  bc_b(ins)   = (ins >> 24) & 0xFF
  bc_c(ins)   = (ins >> 16) & 0xFF
  bc_d(ins)   = (ins >> 16) & 0xFFFF
```

## 7. Frame Layout on Stack

```
[base + framesize - 1]  top slot
...
[base + 0]              first slot (function ref in LJ_FR2 mode)
[base - 1]              frame marker (type/size or PC)
[base - 2 .. base - N]  previous frame metadata
```

Frames linked via `FrameLink`: either `ftsz` (type+size) or `pcr` (PC).
The interpreter walks this chain to unwind frames on return.

## 8. Hotcount — The Low-Overhead Profiling Mechanism

**Design:** 16-bit counter per hashed PC. Decremented by 2 on LOOP, 1 on CALL.
Underflow triggers recording.

```
HOTCOUNT_SIZE = 64       — 64 hash buckets
HOTCOUNT_PCMASK = (63 * sizeof(HotCount))  — hash to bucket
HOTCOUNT_LOOP = 2        — decrement per loop
HOTCOUNT_CALL = 1        — decrement per call
```

**x64 assembly (from vm_x64.dasc):**
```asm
mov   reg, PCd           ; load current PC
shr   reg, 1             ; compensate for 4-byte instruction alignment
and   reg, HOTCOUNT_PCMASK  ; hash to 64 buckets
sub   word [DISPATCH+reg+hot_offset], 2  ; decrement 16-bit counter
jb    ->vm_hotloop        ; underflow → start recording
```

This is 5 instructions embedded directly in the interpreter body. No function
call overhead, no separate profiling thread.

**Why collisions don't matter:** Two different bytecode positions hashing to
the same bucket just means one reaches the hotcount threshold slightly faster.
The recorder will either succeed (good) or abort quickly (penalty slot prevents
repeated attempts). With 16-bit counters, the mechanism is effectively stateless
(overflow wraps harmlessly).

**Lazy disabling:** When JIT is disabled for a prototype (PROTO_NOJIT flag),
the hot bytecode is patched in-place:
```c
setbc_op(J->pc, (int)bc_op(*J->pc) + (int)BC_ILOOP - (int)BC_LOOP);
```
This replaces LOOP→ILOOP, FORL→IFORL, etc. No flag check every iteration.
